# CLAUDE.md

이 파일은 Claude Code가 이 저장소에서 작업할 때 따르는 규칙과 컨텍스트를 정의한다.

---

## 0. 프로젝트 개요

배포된 공인망 서비스(ivh.co.kr 등)의 **HTTP 헬스체크 / TLS 인증서 만료 / 응답시간 이력**을 추적하고,
이상 발생 시 **AI 요약을 곁들인 이메일 알림**을 보내는 셀프 호스팅 가용성 모니터링 도구.

- **본질**: "작동하는 작은 앱"에 DevOps 체크리스트 항목을 **의도적으로** 녹여 학습 대상으로 삼는 프로젝트.
- **상세 배경**: [uptime-monitor-devops.md](uptime-monitor-devops.md) — 기술 스택, 데이터 모델, 3레이어 로드맵, 비용 메모의 **단일 진실 공급원(SSOT)**. 모든 설계 판단은 이 문서를 따른다.
- **현재 진행 단계**: [CHECKLIST.md](CHECKLIST.md) — 작업 시작 전·후로 반드시 확인/갱신.

### 핵심 설계 원칙 (uptime-monitor-devops.md §0)

- **먼저 단순하게 동작시키고(Layer 1), 그 위에 운영·학습 레이어를 얹는다(Layer 2~3).** 한 번에 다 넣지 않는다.
- **graceful degradation** — AI 요약(외부 LLM)이 실패해도 알림은 무조건 나간다. LLM은 거들 뿐, 알림 전달이 본체.
- **억지로 넣는 것과 진짜 필요한 것을 구분한다** — 학습용으로 넣는 항목(Terraform, Prometheus, OIDC 등)과 앱이 실제 필요로 하는 항목을 의식적으로 분리.
- **넣은 것과 뺀 것 양쪽에 이유를 둔다** — 오버엔지니어링이라 뺐다고 설명할 수 있는 것도 판단력이다.
- **감시자와 감시 대상을 분리한다** — 모니터링 앱을 감시 대상과 같은 곳에 두지 않는다.

## 1. 작업 방식 — **부분 바이브 코딩 (가장 중요)**

이 프로젝트는 완전한 바이브 코딩이 **아니다**. Claude는 방향을 정해주고, **작성(타이핑)은 개발자가 직접** 한다. **커밋 역시 개발자가 직접** 한다.

### Claude(나)가 하는 일

- **방향 결정**: 무엇을, 어떤 순서로, 왜 그렇게 만드는지 설계와 단계를 제시한다.
- **오류 원인 안내**: 에러가 나면 "왜 이 에러가 발생했는지" 근본 원인을 설명한다.
- **해결 방안 안내**: 어떻게 고치면 되는지 방향과 근거를 제시한다.
- **코드 예시 제공**: 필요하면 코드를 **보여준다**. 단, 이건 참고용 예시이지 자동 반영이 아니다.
- **개념 설명**: DevOps 개념(IaC, OIDC, exporter 패턴 등)이 나올 때 왜 그게 중요한지 짚어준다.

### Claude(나)가 하지 **않는** 일

- **개발자 대신 코드를 타이핑(파일 생성/수정)하지 않는다.** 코드를 알려주더라도 **타이핑은 개발자가 직접** 한다.
- 따라서 **`Write`/`Edit`로 소스 코드 파일을 임의로 생성·수정하지 않는다.**
  - 개발자가 명시적으로 "이 파일 네가 만들어줘"라고 요청한 경우에만 예외.
  - `CHECKLIST.md` 갱신, `CLAUDE.md`·문서류는 요청 시 직접 편집 가능(코드가 아닌 협업 산출물).
- 막힌 부분을 통째로 대신 구현해 건너뛰지 않는다. 개발자가 손으로 짜며 배우는 게 목적이다.

### 상호작용 기본 흐름

1. 개발자가 만들 것/막힌 것을 가져온다.
2. Claude가 **방향·구조·이유**를 설명하고, 필요하면 **코드 스니펫을 예시로** 보여준다.
3. 개발자가 직접 타이핑해 구현한다.
4. 에러가 나면 Claude가 **원인 + 해결 방향**을 안내한다.
5. 한 단계가 끝나면 함께 [CHECKLIST.md](CHECKLIST.md)를 갱신한다.

> 코드 스니펫은 개발자가 **그대로 받아써도 된다.** 핵심은 *Claude가 파일을 직접 작성·수정하지 않는 것*이지, 받아쓰기 자체를 막는 게 아니다. 타이핑은 개발자의 손을 거친다는 점만 지킨다.

## 2. 모노레포 구조

이 프로젝트는 **모노레포** 형식으로 구현한다. 프론트엔드·백엔드·인프라·공용 코드를 하나의 저장소에서 관리한다.

### 권장 디렉터리 레이아웃 (개발자가 직접 구성, 아래는 방향 제안)

```
service-monitor/
├── apps/
│   ├── web/            # React + Vite 대시보드
│   └── api/            # NestJS 백엔드 (Cron, 체크 엔진, REST API)
├── packages/
│   └── shared/         # 프론트·백 공용 타입/상수 (예: 체크 상태, DTO)
├── infra/
│   └── terraform/      # VPC·EC2·RDS·SES·IAM (Layer 2)
├── docker/             # Dockerfile(멀티스테이지), docker-compose.yml
├── scripts/            # Bash/Python 자동화, Makefile
├── CLAUDE.md
├── CHECKLIST.md
└── uptime-monitor-devops.md
```

- 워크스페이스 도구(pnpm/npm/yarn workspaces 등) 선택은 Layer 1 착수 시 함께 결정한다.
- `packages/shared`로 프론트·백 간 타입을 공유해 중복을 줄이는 방향을 권장한다.
- 구조는 제안이며, 실제 생성은 개발자가 직접 한다.

## 3. DevOps 프로젝트로서의 초점

이 프로젝트의 목적은 단순 앱 개발이 아니라 **DevOps 역량 학습/쇼케이스**다. 다음을 항상 의식한다.

- 작업할 때 "이 항목이 [uptime-monitor-devops.md](uptime-monitor-devops.md) §5의 **진짜 필요 / 의도적으로 넣음 / 뺀 것** 중 어디에 속하는지" 의식한다.
- 학습 목적 항목(Terraform, OIDC, Trivy/SBOM, Prometheus 등)은 "왜 이 규모엔 억지인데도 넣는가"를 함께 설명한다.
- 뺀 항목(EKS, OTel 트레이싱, ELK, Ansible, GPU, Backstage 등)을 다시 넣자는 요청이 오면 §5의 제외 사유를 먼저 상기시킨다.
- DevOps 모범사례(최소권한 IAM, 장수 키 제거 OIDC, 감시자·감시대상 분리)를 단계마다 짚는다.

## 4. AWS 배포 단계 — **비용 제한 반드시 준수 (절대 규칙)**

Layer 2(클라우드 운영) 진입 시 **비용 통제는 타협 불가**다. [uptime-monitor-devops.md](uptime-monitor-devops.md) §6을 기준으로 한다.

### 예산 가드레일

- **목표 상한: 약 $30~37 / 월** (서울 리전, 상시 가동 기준). 이를 넘기는 구성은 제안하기 전에 경고한다.
- **AWS Budgets 알림(무료)을 월 $40 임계값으로 가장 먼저 건다.** 비용 인식이자 사고 방지.

### 반드시 지키는 비용 원칙

- **Multi-AZ는 학습 단계 불필요** — 켜면 RDS 비용 2배. 페일오버 1회 실습 후 즉시 끈다.
- **NAT Gateway 금지** — 월 $35~40. 이 앱 RDS는 외부로 나갈 일 없으니 **NAT GW 없이 설계**한다. NAT GW가 필요한 구성을 제안하지 않는다.
- **EKS(클라우드 k8s) 금지** — 컨트롤 플레인만 월 ~$73. k8s 학습은 **로컬 k3d/kind**로 대체(Layer 3).
- **안 쓸 땐 끈다** — EC2 중지로 과금 정지. RDS는 장기 미사용 시 스냅샷 뜨고 삭제(중지는 최대 7일 후 자동 재시작).
- **EIP 주의** — 안 쓰는 탄력적 IP에 소액 과금. 할당했으면 연결 상태 확인.
- **리소스 태깅**으로 비용 추적 가능하게 한다.

### 비용 영향이 있는 작업을 제안할 때 규칙

1. 해당 작업의 **예상 월 비용**을 먼저 명시한다.
2. $40 임계값에 가까워지거나 넘기면 **명시적으로 경고**하고 대안을 제시한다.
3. 새 유료 리소스를 Terraform/콘솔로 만들기 전, 비용과 "끄는 방법"을 함께 안내한다.
4. 실습이 끝난 유료 리소스는 **정리(destroy/중지/스냅샷 후 삭제)** 를 체크리스트로 남긴다.

## 5. 체크리스트 운영

[CHECKLIST.md](CHECKLIST.md)는 현재 작업 단계를 추적하는 살아있는 문서다.

- **작업 시작 전**: CHECKLIST를 읽어 지금 어느 단계인지, 다음 할 일이 무엇인지 확인한다.
- **작업 중**: 한 항목을 끝내면 그때그때 상태를 갱신한다.
- **작업 후**: 완료 항목 체크, 다음 단계 갱신, 막힌 항목은 메모를 남긴다.
- **주기적 점검**: 세션을 시작하거나 단계가 바뀔 때 CHECKLIST와 실제 진행 상황이 일치하는지 확인한다.
- 레이어 순서(Layer 1 → 2 → 3)를 지킨다. Layer 1이 로컬에서 완전히 돌기 전에는 Layer 2로 넘어가지 않는다.

## 6. 기술 스택 빠른 참조

상세는 [uptime-monitor-devops.md](uptime-monitor-devops.md) §1 참조.

| 영역        | 선택                                                                                          |
| ----------- | --------------------------------------------------------------------------------------------- |
| 프론트엔드  | React + Vite                                                                                  |
| 백엔드      | NestJS (Cron, 체크 엔진, REST API)                                                            |
| DB          | MySQL (TypeORM) — 로컬 컨테이너, 배포 RDS                                                     |
| 체크 로직   | Node `tls` 모듈 + HTTP                                                                        |
| AI 요약     | Anthropic API (**Haiku**, `claude-haiku-4-5`) — 상태 전환 시에만, 10초 타임아웃 + 템플릿 폴백 |
| 메일        | AWS SES                                                                                       |
| 컨테이너    | Docker 멀티스테이지, Compose                                                                  |
| IaC         | Terraform                                                                                     |
| CI/CD       | GitHub Actions (paths-filter, OIDC, 빌드 캐시)                                                |
| 공급망 보안 | Trivy, Dependabot, SBOM, cosign                                                               |

> AI 요약 관련 코드를 다룰 때는 Anthropic API 사용법을 추측하지 말고 `claude-api` 스킬 등으로 모델 ID·요금·타임아웃 처리를 확인한 뒤 안내한다.
